How it works

From install to first run without rewriting CI.

1

Install the app

Add the Speedforge GitHub App to the repositories you choose. Speedforge derives org and installation context from GitHub.

2

Review the migration PRs

Speedforge opens pull requests that switch eligible jobs to runs-on: speedforge. You review, test, and merge.

3

Run on isolated capacity

Each job gets a fresh runner in your org's dedicated pool, with the same Actions logs and workflow controls you already use.

# .github/workflows/ci.yml
jobs:
  build:
    runs-on: speedforge      # was: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: make test

Architecture

Standard GitHub Actions protocol, managed runner substrate.

Control plane

GitHub App events drive onboarding. GitOps reconciles runner scale sets, network policy, and cache substrate from version-controlled source.

Runner plane

Actions Runner Controller starts ephemeral runners on demand. Jobs see normal GitHub Actions behavior while the underlying fleet scales to zero.

Cache plane

Node-local pnpm, uv, Buildx, and Docker Hub pull-through caches improve warm-path runs without changing workflow syntax.

Safety plane

Repository-scoped installs, isolated namespaces, private runner pools, and public-repo guardrails keep tenant boundaries explicit.
